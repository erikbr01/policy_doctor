# DAgger Interactive Evaluation for Robomimic-Based Environments

Complete guide for running DAgger-style human-in-the-loop demonstrations with automatic intervention timing based on behavior graph V-values.

Supports any robomimic-compatible environment: **robomimic** (kitchen), **robocasa**, **libero**, **blockpush**, **mimicgen**, etc.

## Overview

This system combines three key components:

1. **Behavior Graph Monitoring** — classifies each policy timestep into behavioral clusters and computes Bellman V-values to detect low-confidence states
2. **Interactive Environment** — records per-step data with `acting_agent` labels (robot or human) for DAgger retraining
3. **Keyboard Controller** — enables human teleoperation when the behavior graph flags a problematic trajectory

The workflow:
```
1. Policy rolls out in "robot mode"
2. MonitoredPolicy.predict_action() classifies each step
3. If behavior graph V(node) < threshold → auto-trigger intervention
4. Human takes over (press Space), provides corrective actions
5. Episode saved with per-step (robot|human) labels
6. Collected human trajectories merged into training dataset
7. Retrain policy on D_0 ∪ D_1 ∪ ... (original + human demos)
```

## Quick Start

### 1. Set Up Environment (macOS M4)

```bash
cd /path/to/policy_doctor

# Create conda environment
bash scripts/setup_dagger_env.sh
conda activate policy_doctor_dagger
```

The script installs:
- PyTorch 2.x with MPS support
- robosuite + robocasa
- infembed for influence scoring
- pynput for keyboard input

**Note**: On first run, you'll be prompted for **Accessibility permission** (macOS requires this for keyboard monitoring). Grant it in System Preferences → Security & Privacy → Accessibility.

### 2. Prepare Monitoring Artifacts

Before running DAgger, you need:
- Trained policy checkpoint: `checkpoints/best.ckpt`
- InfEmbed fit: `infembed_fit.pt` (from `compute_infembed_embeddings.py`)
- InfEmbed embeddings: `infembed_embeddings.npz`
- Behavior graph clustering: `clustering/` directory

All generated by the standard `policy_doctor` pipeline (see `CLAUDE.md`).

### 3. Run DAgger Episodes

Use the generic `run_dagger.py` script (works for all environments):

**Kitchen Square Stacking:**
```bash
python scripts/run_dagger.py \
  --task square_mh \
  --train_dir third_party/cupid/data/outputs/train/square_mh/2024-04-20_xyz \
  --train_ckpt best \
  --infembed_fit /path/to/eval/infembed_fit.pt \
  --infembed_npz /path/to/eval/infembed_embeddings.npz \
  --clustering_dir /path/to/eval/clustering/run_001 \
  --output_dir /tmp/dagger_square_001 \
  --num_episodes 5 \
  --intervention_threshold 0.0 \
  --device auto
```

**Robocasa (Pick-and-Place):**
```bash
python scripts/run_dagger.py \
  --task robocasa_layout_lowdim \
  --train_dir third_party/cupid/data/outputs/train/robocasa_layout_lowdim/2024-04-20_xyz \
  --train_ckpt best \
  --infembed_fit /path/to/eval/infembed_fit.pt \
  --infembed_npz /path/to/eval/infembed_embeddings.npz \
  --clustering_dir /path/to/eval/clustering/run_001 \
  --output_dir /tmp/dagger_robocasa_001 \
  --num_episodes 5 \
  --device auto
```

**Kitchen Transport:**
```bash
python scripts/run_dagger.py \
  --task transport_mh \
  --train_dir third_party/cupid/data/outputs/train/transport_mh/... \
  ... (same options as above)
```

**Kitchen Lift:**
```bash
python scripts/run_dagger.py \
  --task lift_mh \
  --train_dir third_party/cupid/data/outputs/train/lift_mh/... \
  ... (same options as above)
```

The `--task` parameter automatically infers the dataset path. For custom dataset locations, pass `--dataset_path /path/to/dataset.hdf5`.

### 4. Convert to Training Dataset

After collecting episodes:

```bash
python scripts/build_dagger_dataset.py \
  --episodes_dir /tmp/dagger_run_001 \
  --output_hdf5 /tmp/kitchen_lowdim_with_dagger.hdf5 \
  --filter_human_only
```

This extracts human-controlled segments and writes to robomimic HDF5 format.

### 5. Retrain Policy

```bash
# Fine-tune on merged dataset
python -m policy_doctor.scripts.run_pipeline \
  steps=[train_curated] \
  task_config=robocasa_layout_lowdim \
  ++data_source.baseline.diffusion_dataset_path=/tmp/kitchen_lowdim_with_dagger.hdf5
```

## Keyboard Controls

During a DAgger episode:

| Key | Action |
|-----|--------|
| **Space** | Toggle human/robot control (press once to take over, again to release) |
| **W** | Arm move up (+z) |
| **S** | Arm move down (-z) |
| **A** | Arm move left (-x) |
| **D** | Arm move right (+x) |
| **Q** | Arm move backward (-y) |
| **E** | Arm move forward (+y) |
| **G** | Gripper close |
| **H** | Gripper open |
| **I** | Base forward |
| **K** | Base backward |
| **J** | Base turn left |
| **L** | Base turn right |
| **Ctrl+C** | Stop all episodes |

## Architecture & Key Components

### `RobomimicDAggerEnv` (`policy_doctor/envs/robomimic_dagger_env.py`)

Generic wrapper around MultiStepWrapper(RobomimicLowdimWrapper(...)) that works with any robomimic-compatible environment (robomimic, kitchen, robocasa, libero, blockpush, mimicgen, etc.):

```python
env = RobomimicDAggerEnv(
    inner_env=multistep_wrapper,
    obs_keys=["object", "robot0_eef_pos", ...],
    output_dir="/tmp/dagger",
)

# Record per-step data with labels
for t in range(num_steps):
    obs, reward, done, info = env.step(action)

# Save episode as pkl
path = env.save_episode()  # ep0000.pkl
```

**Data recorded per step:**
- `obs`: dict with per-key observations (for HDF5 conversion)
- `stacked_obs`: concatenated observation (n_obs_steps, obs_dim)
- `action`: action executed
- `reward`, `done`: env feedback
- `acting_agent`: "robot" or "human" (for filtering)
- `sim_state`: MuJoCo state (for deterministic replay)

### `KeyboardInterventionDevice` (`policy_doctor/envs/intervention_device.py`)

Keyboard-based teleoperation (pynput-based):

```python
device = KeyboardInterventionDevice(action_dim=10, step_size=0.05)

# Get human action if keys are pressed
human_action = device.get_action()  # np.ndarray(10,) or None

# Check if human is controlling
if device.is_intervening:
    # Proceed with human action
    pass
```

Composes actions by summing key bindings. Mid-step interrupt support (check `device.is_intervening` during action chunk execution).

### `RobocasaDAggerRunner` (`policy_doctor/envs/dagger_runner.py`)

Main control loop:

```python
runner = RobocasaDAggerRunner(
    monitored_policy=monitored_policy,    # wrapped policy with behavior graph
    env=env,                              # data-recording env
    intervention_device=device,           # keyboard input
    n_obs_steps=2,
    n_action_steps=8,
    max_steps=500,
    output_dir="/tmp/dagger",
    visualizer=visualizer,                # optional live display
)

records = runner.run(n_episodes=5)
```

**Control flow per episode:**
1. Robot mode: policy rolls out for `n_action_steps`
2. After each chunk: check if `NodeValueThresholdRule` triggered → switch to human
3. Also check `device.is_intervening` → manual override
4. Human mode: wait for keyboard input, execute human actions
5. Return control when `device.is_intervening` flips back
6. Save episode with per-step labels

### `MonitoredPolicy` Integration

The `MonitoredPolicy` (existing, from `policy_doctor/monitoring/monitored_policy.py`) wraps any policy:

```python
monitored_policy = MonitoredPolicy(
    policy=policy,
    classifier=classifier,                # TrajectoryClassifier
    intervention_rule=NodeValueThresholdRule(node_values, threshold=0.0),
)

# After each predict_action() call:
result = monitored_policy.predict_action(obs_dict)
intervention_decision = monitored_policy.episode_results[-1]["intervention"]

if intervention_decision.triggered:
    print(f"Intervention: {intervention_decision.reason}")
    # (e.g., "value=-0.2345 < threshold=0.0000")
```

## Data Format & Retraining

### Collected Episodes Format

Each episode pkl (e.g., `ep0000.pkl`) is a pandas DataFrame with columns:

| Column | Shape | Note |
|--------|-------|------|
| `timestep` | () | Step number in episode |
| `obs` | dict | Per-key obs: `{object: (8,), robot0_eef_pos: (3,), ...}` |
| `stacked_obs` | (2, 23) | Concatenated obs (n_obs_steps, obs_dim) |
| `action` | (10,) | Action executed |
| `reward` | () | Environment reward |
| `done` | () | Episode termination flag |
| `acting_agent` | str | "robot" or "human" |
| `sim_state` | (30,) | MuJoCo qpos+qvel for replay |

### Converting to HDF5

The `build_dagger_dataset.py` script reads pkl files and writes robomimic HDF5:

```
data/
  demo_0/
    obs/
      object (T, 8)
      robot0_eef_pos (T, 3)
      robot0_eef_quat (T, 4)
      robot0_gripper_qpos (T, 2)
    actions (T, 10)
    rewards (T,)
    dones (T,)
    states (T, 30)
    acting_agent (T,)    # custom: 1=human, 0=robot
  demo_1/
    ...
```

Two modes:
- **`--filter_human_only`**: Extract only contiguous human-controlled segments (standard HG-DAgger)
- **Full episodes**: Include entire rollout for offline analysis

## Device Support & macOS

### Auto Device Detection

All `device='auto'` parameters (CLI and Python API) auto-detect:

1. **CUDA**: `torch.cuda.is_available()` → use `cuda:0`
2. **Apple Silicon (MPS)**: `torch.backends.mps.is_available()` → use `mps`
3. **CPU fallback**: `cpu`

Useful for cross-platform scripts.

### MPS-Specific Notes

- PyTorch 2.x required (for `torch.func` and MPS)
- InfEmbedStreamScorer runs on MPS without issues
- MuJoCo: use modern `mujoco` package (DeepMind), not `free-mujoco-py` (limited ARM support)
- Keyboard listener (pynput) needs Accessibility permission on macOS

## Troubleshooting

### "No module named pynput"
→ Make sure to activate the `policy_doctor_dagger` environment and complete setup.

### "Accessibility permission denied" (macOS)
→ Grant permission: System Preferences → Security & Privacy → Accessibility → allow Terminal (or your IDE).

### "device='cuda:0' not available on macOS"
→ Use `--device auto` or `--device mps`. The CLI defaults to auto-detection.

### "Cannot import torch.func"
→ Ensure PyTorch 2.x is installed. The conda env uses 2.2.0.

### "MuJoCo rendering fails"
→ Rendering is offscreen (no native viewer window). Camera images displayed in matplotlib. If matplotlib fails, use `--no_visualization`.

### "Running out of GPU memory (MPS)"
→ Reduce batch size, reduce `n_obs_steps` or `n_action_steps`, or use `--device cpu`.

## Example Workflows

### Kitchen Square Stacking (Square_MH)

```bash
# 1. Train baseline policy (existing pipeline)
python -m policy_doctor.scripts.run_pipeline \
  steps=[train_curated] task_config=square_mh_jan28

# 2. Compute InfEmbed artifacts for monitoring
# (Done by pipeline attribution step)

# 3. Collect human demonstrations

  --task square_mh \
  --train_dir <checkpoint_dir> \
  --infembed_fit <eval_dir>/infembed_fit.pt \
  --infembed_npz <eval_dir>/infembed_embeddings.npz \
  --clustering_dir <eval_dir>/clustering/run_001 \
  --output_dir /tmp/dagger_square_round1 \
  --num_episodes 10

  --episodes_dir /tmp/dagger_square_round1 \
  --output_hdf5 data/source/kitchen_square_dagger_r1.hdf5 \

python -m policy_doctor.scripts.run_pipeline \
  steps=[train_curated] task_config=square_mh_jan28 \
  ++baseline.diffusion_dataset_path=data/source/kitchen_square_dagger_r1.hdf5

```
